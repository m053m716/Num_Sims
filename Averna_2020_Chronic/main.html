Main Objective

The goal of this simulation is to understand what happens when we draw many repeated surrogate samples from a fixed set of sample instants, wherein a single element of each surrogate is drawn from the total distribution of sample times with a uniform random chance of any sample time being selected. In particular, we would like to characterize how similar such a surrogate is to a Poisson process with parameter λ, in this case, where λ defines the total number of elements drawn from the fixed time window.

Processing

Processing (and particularly smoothing) has a large effect on the data. This is because any smoothing effectively applies a low-pass filter to the data, which by its definition, causes consecutive data samples to become more correlated (provided that there is frequency content present that is outside of the realized passband).

Smooth rates

The end-goal is to obtain X_s_shuff, the smoothed instantaneous firing rate estimates. At this step the choice of a smoothing parameter is somewhat important and something that may differ from group to group. There are many ways to smooth the data; in MATLAB, a simple choice is to implement a Savitzky-Golay filter, which effectively interpolates a polynomial (order parameter) over a sliding window with an odd number of samples (framelen parameter). Both of these parameter choices influence the degree of smoothing, which to some extent also depends on the data; an alternative way is to use a lowpass filter with a fixed cutoff frequency, which makes it easier to replicate the filter that was applied in each instance. For our purposes, it should suffice that a larger framelen makes the data "smoother" while a shorter one allows "faster fluctuations" (qualitatively).
  • We leave selection of the wlen parameter up to a slider to make it easier to see how it affects things.
  • The user can also change the polynomial fit order.
  • For simplicity, we fix weights (Hamming window).

Estimate at each sample

Note that we have recovered X_s_shuff, which is the shuffled spike rates; however, for the purposes of comparing the observed surrogates to those generated by a Poisson process defined by λ, we need to recover the rate estimate at each time sample. We do this in two steps:
  1. Constrain any smoothing to a non-negative value (the sgolayfilt function may interpolate to negative values, particularly at the edges, depending on the optimal fit of the polynomial in that particular region).
  2. Divide each time sample by its sample period to get the approximate instantaneous rate estimate, for that time period.

Compute the endpoint output: Fano Factor

We now have the matrix of recovered rate estimates, lambda_hat. We now would like to test against the null hypothesis, which is that the simulated data does not deviate substantially from a Poisson process P defined by λ. It is evident from how we simulated these data that the mean of this process over the 1-second epoch surrogates we have created is λ, by definition.
P should meet the criterion: , which in english means that the Poisson process defined by λ should have a mean equal to its variance. Deviations from this definition are characterized by the Fano Factor and can give us an indicator for whether the results obtained by surrogate generation could be reasonably approximated using an analytical approach.

Visualize Results

To visualize the simulation results, we should just plot a histogram that is the distribution of the Fano Factor obtained when we shuffle the data like this. In the default case, we will have 100 elements for time-series mean and variance, and 1000 elements for cross-trial mean and variance.
In the ideal case for Poisson process P described above, the Fano Factor distribution should approximately be centered on 1 for both cases. Note that even setting a very-short frame length for the sgolayfilt has a very strong influence on the distribution of the output Fano Factors. In general, smoothing the data decreases the Fano Factor, causing the apparent result to become underdispersed, or more uniform. Intuitively, this makes sense, as we are "smoothing" the data and therefore causing large, fast changes to go away.
